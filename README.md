# 微信好友列表提取工具

Windows 下从本地微信数据库提取联系人 / 好友列表，导出为 CSV。

> 仅用于导出**你自己账号**的本地数据（例如备份通讯录）。请勿用于未经授权访问他人数据，使用须遵守当地法律法规。

---

## 环境要求

- **系统**：Windows（依赖 `ReadProcessMemory` / `VirtualQueryEx`，不支持 macOS / Linux）
- **Python**：3.7+
- **微信**：PC 版 4.x，基于 SQLCipher 4 加密。依赖“密钥以 `x'<hex>'` 文本格式驻留内存”这一特征：**4.1.9.x 实测可用；4.1.10+ 改了存储方式不可用**（详见下方“故障排查”的版本兼容性）
- **依赖**：

  ```bash
  pip install -r requirements.txt
  ```

---

## 快速开始（三步）

### 1. 改配置

打开 `config.py`，改两个地方：

```python
DB_DIR = r"D:\xwechat_files\wxid_xxxxxxxx\db_storage"  # 你的数据库目录
ACCOUNT_NAME = "我的微信"                                # 账号名，用于命名好友列表文件
```

> `DB_DIR` 路径在微信「设置 → 通用 → 文件管理」可查到，进到 `xwechat_files\<wxid>\db_storage` 即可。
> `ACCOUNT_NAME` 随便填好认的名字（如微信号、备注），它只用来给导出的好友列表文件命名，例如填 `txsp668` 就生成 `txsp668的好友列表.csv`。

### 2. 提取密钥（管理员身份）

保持微信登录，**以管理员身份**打开终端运行：

```bash
python extract_key.py
```

成功后密钥打印在终端，并保存到 `key.txt`。

### 3. 解密导出

```bash
python export_contacts.py
```

一次生成**两份 CSV**，输出到配置的 `OUTPUT_DIR`：

- `contacts.csv` —— **全量版**：全部联系人（含公众号/群聊/系统账号）+ 全部原始字段（英文表头）
- `<账号名>的好友列表.csv` —— **精简版**：仅真实好友（local_type=1）+ 中文表头 + 手机号

另有中间产物 `contact_dec.db`（解密后的标准 SQLite 库，可用 [DB Browser for SQLite](https://sqlitebrowser.org/) 直接打开）。

> `<账号名>` 取自 `config.py` 的 `ACCOUNT_NAME`。例如填 `txsp668`，精简版就是 `txsp668的好友列表.csv`。

## 运行前提（务必先看，否则大概率失败）

1. **必须以管理员身份运行终端**。读取微信进程内存需要管理员权限，普通权限下 `OpenProcess` 会失败、提取不到密钥。
   - 做法：开始菜单搜索 `PowerShell` 或 `命令提示符` → 右键 → **以管理员身份运行**，再 `cd` 到项目目录跑脚本。
   - **直接双击 `.py` 文件是普通权限，必然失败。**

2. **微信要处于已登录运行状态**。密钥只有在微信运行时才驻留内存；没登录或刚启动还没解密数据库时，扫不到密钥。

3. **可能需要临时关闭杀毒软件 / Windows Defender**。本工具用 `ReadProcessMemory` 读取其它进程内存，这个行为常被杀软拦截，甚至直接把脚本/打包后的 exe 报成木马。如果脚本疑似被拦（无报错却读不到任何内存、或文件被自动删除），临时关闭实时防护再试。

## 故障排查

| 现象 | 可能原因 / 处理 |
|------|----------------|
| `未找到密钥` | ① 不是管理员身份运行 → 用管理员开终端；② 微信没登录 → 先登录；③ **微信版本不支持**（见下方表格）→ 回退到 4.1.9.x |
| `Weixin.exe 未运行` | 微信没开，或进程名不是 `Weixin.exe`（老版本是 `WeChat.exe`，改 `config.py` 的 `WECHAT_PROCESS`） |
| `找不到 contact.db` | `config.py` 的 `DB_DIR` 路径不对，或填到了别的账号的 wxid |
| `HMAC 校验失败` | 密钥与该数据库不匹配（多账号时拿错了 key），或数据库文件损坏 |
| 脚本无报错但读不到任何内存 / 文件被删 | 被杀软拦截，临时关闭实时防护重试 |
| `ModuleNotFoundError: Crypto` | 没装依赖，跑 `pip install -r requirements.txt`（注意是 `pycryptodome`，不是 `pycrypto`） |

**微信版本兼容性**（关键）：本工具依赖“密钥以 `x'<hex>'` 文本格式存在内存”这一特征。

| 微信版本 | 是否可用 |
|----------|----------|
| 4.1.9.x | ✅ 可用 |
| 4.1.10.x | ❌ 改了密钥内存存储方式，扫不到，需回退到 4.1.9.x（详见下方“遇到的问题”） |

查当前微信版本：

```powershell
(Get-Item (Get-Process Weixin | Sort WorkingSet64 -Desc | Select -First 1).Path).VersionInfo.FileVersion
```

> **下载微信历史版本**：如果当前版本是 4.1.10+，需要回退到 4.1.9.x 才能用。可在 <https://wechat-pc.cn.uptodown.com/windows/versions> 下载微信 PC 版的历史安装包。回退后建议**关闭微信自动更新**，否则会再次升级到不可用的版本。

## 导出字段

运行后得到两份 CSV：

**全量版 `contacts.csv`** —— 数据库 `contact` 表的全部记录和全部原始字段（`id` / `username` / `local_type` / `alias` / `nick_name` / `extra_buffer` 等 22 列，英文表头）。`extra_buffer` 等二进制字段以原始 bytes 形式输出。适合需要完整数据做进一步处理的场景。

**精简版 `<账号名>的好友列表.csv`** —— 只保留真实好友（local_type=1）和常用列，中文表头：

| 字段 | 说明 |
|------|------|
| 序号 | 自动编号 |
| 微信号 | username，登录账号 |
| 类型 | local_type（1=好友，2=群聊，3=公众号，0/5=系统账号） |
| 微信号别名 | alias，自定义微信号 |
| 备注名 | remark |
| 昵称 | nick_name |
| 手机号 | 从 extra_buffer 解析，部分好友有 |

> 提示：`local_type` 一般 1=好友、2=群聊、3=公众号、0/5=系统账号，但**有些系统账号（如 `medianote` 语音记事本、`floatbottle` 漂流瓶）也被标成了 type=1**，因此会混进精简版“好友”列表。需要更干净的列表可在导出后按微信号剔除。

## 原理简述

微信本地库用 **SQLCipher 4** 加密：4096 字节一页、AES-256-CBC、PBKDF2-SHA512 派生密钥、每页 HMAC-SHA512 校验，文件无法直接读。

但微信运行时必须解密数据库，**密钥就在进程内存里**，且以 `x'<64位十六进制>'` 的明文文本形式存放。本工具：

1. 用 `VirtualQueryEx` 枚举微信进程的可读内存
2. 正则匹配 `x'<hex>'` 候选密钥
3. 用数据库第一页做 HMAC-SHA512 校验，确认是真密钥
4. 拿到密钥后逐页 AES 解密，还原成标准 SQLite 文件

> 若微信新版本改变了密钥在内存中的存储方式（例如不再是 `x'<hex>'` 文本），`extract_key.py` 的匹配逻辑就需要相应调整。

## 深入原理

### 数据库是怎么加密的

微信把联系人存在本地的 `contact.db`，这是一个用 **SQLCipher 4** 加密的 SQLite 数据库：

- 整个文件按页（4096 字节/页）分块，每页独立加密
- 加密算法：**AES-256-CBC**
- 密钥派生：用户密钥 + 文件头随机 salt，经 **PBKDF2-SHA512** 派生出 256 位加密密钥
- 完整性校验：每页末尾附 **HMAC-SHA512** 签名，防篡改
- 第 1 页头部 16 字节是明文 salt，其余全部加密

所以只拿到加密文件、不知道密钥，是无法直接读取的。

### 破解思路：从内存里取密钥

文件加密无法暴力破解，换个方向——**微信运行时必须解密数据库，密钥一定在内存里**。

微信登录后会把解密密钥（32 字节）加载进进程内存，以十六进制字符串形式存放，格式为 `x'<64个十六进制字符>'`，且在进程运行期间一直驻留、不主动清除。

### 密钥提取：逐步拆解

**第一步：定位微信进程**

```
tasklist /FI "IMAGENAME eq Weixin.exe"
```

找到 PID，选内存占用最大的主进程（密钥通常在主进程里）。

**第二步：枚举可读内存区域**

用 `VirtualQueryEx` 遍历所有内存页，筛出状态 `MEM_COMMIT`、权限可读（`PAGE_READONLY` / `PAGE_READWRITE` 等）的区域，跳过超大块（>500MB）避免太慢。

**第三步：扫描内存，搜索密钥格式**

正则匹配：

```
x'([0-9a-fA-F]{64,192})'
```

匹配到的十六进制串两种情况：

- **64 位**：只有密钥，salt 从数据库文件头读取
- **96 位**：前 64 位是密钥，后 32 位是 salt（拼在一起存）

**第四步：HMAC 验证，确认是真实密钥**

候选密钥不能直接用，要用它对数据库第一页做 HMAC-SHA512 校验：

```
mac_salt  = salt XOR 0x3A（逐字节）
mac_key   = PBKDF2-SHA512(候选密钥, mac_salt, 迭代2次, 长度32字节)
hmac_data = page1[16 : 4032]   （去掉头部 salt，保留到含 IV）
计算 HMAC-SHA512(mac_key, hmac_data + 小端页号(1))
与 page1[4032:4096] 对比
```

一致 → 密钥正确。

> **注意这里有两个不同的 PBKDF2，别混淆**：① 派生**主加密密钥**用 64000 次迭代（见“数据库是怎么加密的”）；② 这里派生 **HMAC 校验用的 mac_key** 只迭代 2 次。两者用途不同、迭代次数不同。而且本工具从内存直接拿到的已经是①的成品密钥，所以实际解密时①那步也省了（见下方“关于 PBKDF2”）。

### 数据库解密：逐页还原

拿到真实密钥后逐页解密：

```
对每一页：
  IV   = page[4016:4032]（每页末尾 80 字节保留区的前 16 字节）
  明文 = AES-256-CBC解密(密钥, IV, 密文部分)

第 1 页特殊处理：
  密文范围 = page[16 : 4016]（跳过头部 16 字节 salt）
  解密后在开头补上 SQLite 魔数头 "SQLite format 3\x00"
```

所有页拼合写入新文件，就是一个标准 SQLite 数据库，可用任意工具打开。

> **关于 PBKDF2**：标准 SQLCipher 的 AES 密钥要由“密码 + salt 经 PBKDF2-SHA512 迭代 64000 次”派生。但从内存里取到的 `enc_key` 已经是派生**之后**的最终密钥，所以代码（`crypto_utils.py`）解密时直接用它，不再做 PBKDF2。读代码发现“没有 64000 次迭代”是正常的，不是 bug。

### 为什么能成功

1. **密钥不混淆**：微信没对密钥做额外保护，直接以可搜索的十六进制字符串存在内存里
2. **进程内存可被同权限进程读取**：有管理员权限，`ReadProcessMemory` 就能读
3. **密钥生命周期长**：登录后一直驻留内存，不轮换、不清除

## 遇到的问题与排查记录

### 1. 微信版本决定能否提取（最关键的坑）

密钥提取依赖“内存里有 `x'<hex>'` 文本格式的 key”这一特征，**它和微信版本强相关**：

| 微信版本 | 结果 |
|----------|------|
| **4.1.9.x** | ✅ 正常。内存里能扫到 `x'<hex>'`，HMAC 验证通过，提取成功 |
| **4.1.10.27** | ❌ 失败。改了密钥的内存存储方式，扫遍全部 `Weixin.exe` 进程（实测 5 个进程共 1.2GB 内存）**完全找不到** `x'<hex>'` 文本格式，提取 0 个密钥 |

排查 4.1.10.27 时还验证过：raw key 也**不是**“连续 32 字节明文 + 标准 SQLCipher4 布局”——扫了 665MB、1.7 亿个对齐候选、用完整 SQLite header 强特征校验，仍 0 命中。说明该版本改动了密钥在内存里的形态，旧的文本匹配思路整个失效。

**应对**：提取失败时**先查微信版本**，别一上来怀疑权限/路径。

```powershell
(Get-Item (Get-Process Weixin | Sort WorkingSet64 -Desc | Select -First 1).Path).VersionInfo.FileVersion
```

若是 4.1.10.x，应急办法是**回退微信到 4.1.9.x**（实测回退后立即恢复可用）。历史版本安装包可从 <https://wechat-pc.cn.uptodown.com/windows/versions> 下载。彻底支持 4.1.10 需逆向其新密钥存储格式，工作量较大。

### 2. 误报：HMAC / header 校验不能放太松

逆向 4.1.10 时曾把验证放宽到“解密后只比对 2 字节 page size = `0x1000`”，结果在 15 万候选里偶然撞中一个全是 `00`/指针的假 key（概率约 1/65536），只“通过”了 1 个分片。

**教训**：微信用**同一个 key 加密所有库**，真 key 必须对**多个不同 salt 的库**都校验通过。验证要么用完整 HMAC-SHA512，要么用完整 8 字节 SQLite header 特征（`10 00 | ver ver | 50 40 20 20`），再叠加“跨库复验”，才能杜绝误报。

### 3. HMAC 验证的两个易错细节

正确参数见 `crypto_utils.py` 的 `verify_enc_key`：

- `hmac_data = page1[16 : 4096-80+16]` = `page1[16:4032]`，**末尾要含保留区前 16 字节 IV**；少算这 16 字节会验证失败
- `mac_key` 的 `dklen=32`，不是 64
- `stored_hmac = page1[4032:4096]`（最后 64 字节），HMAC 数据末尾还要再拼上小端页号 `struct.pack("<I", 1)`

### 4. 解密时的 IV 与首页特殊处理

- 每页 IV 在末尾保留区（80 字节）的前 16 字节：`page[4016:4032]`
- 第 1 页跳过开头 16 字节 salt 再解密，解出后补回魔数头 `SQLite format 3\x00`
- `local_type` 含义：1=普通好友，2=群聊，3=公众号，0/5=系统账号

### 5. 跨平台 / 参考实现可得性

- 本工具仅 Windows（依赖 `ReadProcessMemory` / `VirtualQueryEx`）
- 同类参考实现里，`PyWxDump`、`wechat-dump-rs` 已被腾讯 DMCA / 律师函下架，GitHub 直连和 raw 都拿不到核心代码；`ylytdeng/wechat-decrypt`（本项目思路来源）目前 git clone 仍可用
